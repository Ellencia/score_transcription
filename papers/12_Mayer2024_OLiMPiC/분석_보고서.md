# Practical End-to-End Optical Music Recognition for Pianoform Music 분석 보고서

## 핵심 요약

이 논문은 **광학 악보 인식(Optical Music Recognition, OMR)** 을 다룬다. OMR은 **오디오(소리)가 아니라 악보 이미지/사진을 입력으로 받아 디지털 악보로 변환**하는 과제다. 즉, 종이에 인쇄되거나 스캔된 악보 그림을 컴퓨터가 편집·재생 가능한 기호적 형식(MusicXML 등)으로 바꾸는 "악보 사진 → 디지털 악보" 변환이다. 음원에서 음표를 뽑는 자동 음악 채보(AMT)와는 입력 자체가 완전히 다르다.

저자들(체코 카렐 대학 UFAL 그룹)은 특히 어려운 **피아노 악보(pianoform)** 를 표적으로 삼는다. 피아노 악보는 위·아래 두 단(grand staff)에 여러 성부(voice)가 동시에 얽혀 있어, 단순히 왼쪽에서 오른쪽으로 한 줄로 읽어 내려가는 단선율(monophonic) 악보와 달리 선형 시퀀스로 곧장 바꾸기가 어렵다. 그래서 기존 end-to-end OMR 연구들은 제각각의 비표준 인코딩(custom linearized encoding)을 만들어 썼고, 그 결과물이 산업 표준 포맷과 호환되지 않아 실용성이 떨어졌다.

이 논문의 기여는 네 가지다. (1) **LMX(Linearized MusicXML)**: 산업 표준인 MusicXML과 최소 손실로 양방향 변환되는 선형(1차원 토큰 시퀀스) 포맷을 정의해, 이미지→시퀀스(img2seq) 모델을 바로 학습할 수 있게 했다. (2) **OLiMPiC 데이터셋**: OpenScore Lieder 코퍼스를 기반으로 한 합성(synthetic) 및 스캔(scanned) 피아노 악보 데이터셋을 구축했다. (3) **Zeus 모델**: 이 데이터셋에 대한 최신 수준의 베이스라인 img2seq 모델을 학습했다. (4) **TEDn 평가지표**: 트리 편집 거리(tree-edit-distance) 기반 평가지표를 재구현·정비해, OMR 결과의 음악적 정확성을 정량 비교했다. 코드와 데이터는 모두 공개되었다([github.com/ufal/olimpic-icdar24](https://github.com/ufal/olimpic-icdar24)).

## 상세 요약

OMR의 오랜 난제는 "악보는 본질적으로 2차원인데, 현대 신경망 시퀀스 모델은 1차원 토큰 열을 다룬다"는 불일치다. 단선율 악보는 시간순으로 음표를 한 줄로 늘어놓을 수 있어 문제가 작지만, 피아노 악보는 두 보표에 여러 성부가 동시에 진행되므로 "어느 순서로 토큰을 늘어놓을 것인가"가 모호하다. 기존 연구들은 이를 우회하려고 독자적 인코딩을 발명했지만, 그 결과는 MuseScore·Finale·Sibelius 같은 실제 악보 소프트웨어가 읽는 **MusicXML** 표준과 호환되지 않았다. 변환기를 별도로 짜야 하고, 표준이 표현하는 풍부한 음악 정보를 잃는 문제가 생긴다.

저자들의 답은 **새 인코딩을 발명하지 않는 것**이다. LMX는 "MusicXML을 최소한으로 선형화한 것일 뿐, 새로운 표기법이 아니다"라는 설계 원칙을 명시한다. 토큰의 순서는 MusicXML 요소의 등장 순서로 결정되고, 토큰 이름은 MusicXML 요소·값 이름에서 따온다. 또한 LMX는 **소리(audio)가 아니라 시각적 표기(grapheme)** 에 집중한다. 예컨대 음 길이를 `<duration>`(재생용 수치) 대신 `<type>`(악보에 보이는 음표 모양: 4분음표, 8분음표 등)으로 인코딩하고, 음높이 변화(`<alter>`)는 조표와 임시표에서 복원 가능하므로 무시한다. 이렇게 해서 마디 단위로 독립적이며(measure-independent) ML 모델이 학습하기 좋은 토큰 열을 만든다.

데이터셋 **OLiMPiC**는 두 갈래다. **합성(synthetic)** 버전은 MuseScore로 OpenScore Lieder 악보를 렌더링해 만든 깨끗한 이미지로 train/dev/test를 모두 제공한다. **스캔(scanned)** 버전은 실제 인쇄·스캔된 악보로 dev/test만 제공하며, Inkscape를 이용한 수작업 바운딩 박스 주석으로 시스템(악보 한 줄, 대략 4마디) 단위로 정렬했다. 모델은 한 번에 악보 한 줄(시스템)을 입력받아 LMX 토큰 열을 출력하는 img2seq 구조이며, 베이스라인 모델 이름이 **Zeus**다.

평가는 **TEDn(Tree Edit Distance on notation)** 으로 한다. 이는 악보를 트리로 보고, 예측 트리를 정답 트리로 바꾸는 데 필요한 편집 연산 수를 거리로 삼는 지표로, Hajič jr. 등이 ISMIR 2016에서 제안한 것을 이 논문이 표준 파이썬 구현으로 정비했다. TEDn은 계산 비용이 매우 커서 시스템(약 4마디) 단위로만 평가하며, 한 시스템 평가에 30~120초가 걸리고 트리 크기에 따라 대략 제곱으로 증가한다고 명시되어 있다.

## 방법론과 데이터

| 항목 | 내용 |
| --- | --- |
| 과제 유형 | **OMR**: 악보 이미지/스캔 → 디지털 악보 (오디오 아님) |
| 입력 | 피아노 악보(grand staff)의 한 시스템(악보 한 줄, 약 4마디) 이미지 |
| 출력 | LMX(Linearized MusicXML) 토큰 시퀀스 → MusicXML로 역변환 |
| 핵심 포맷 | LMX: MusicXML을 최소 손실로 1차원화, `<type>` 사용·`<alter>` 무시, 마디 독립 |
| 데이터셋 | OLiMPiC (OpenScore Lieder 기반): synthetic(train/dev/test), scanned(dev/test) |
| 비교 데이터셋 | GrandStaff (+ 저자들이 만든 GrandStaff-LMX 확장) |
| 모델 | Zeus (img2seq, SOTA 베이스라인) |
| 평가지표 | TEDn (tree-edit-distance, 시스템 단위 평가, 1시스템당 30~120초) |
| 라이선스 | 코드 MIT, 데이터셋·모델 CC BY-SA |

| 항목 | 값/결과 | 출처/비고 |
| --- | --- | --- |
| LMX ↔ MusicXML 변환 손실 | 최소(minimal losses), 양방향 변환 | 설계 목표이자 검증된 특성 |

> **미검증 주의**: 본 보고서는 arXiv/Springer 정식 PDF의 본문 표(Zeus의 구체적 TEDn 점수, synthetic vs. scanned 성능 격차 등)를 직접 열람하지 못했다. 위 수치는 공개 GitHub 저장소(README, `docs/linearized-musicxml.md`)와 논문 초록·DBLP 서지에서 확인된 사실에 기반하며, 정량 성능 표의 실제 숫자는 Springer 본문에서 추가 확인이 필요하다.

의의는 OMR을 "학술 데모"에서 "실용 도구"에 한 걸음 가깝게 만든 데 있다. 핵심은 **표준 호환성**이다. 독자적 인코딩 대신 MusicXML과 호환되는 LMX를 씀으로써, OMR 모델의 출력이 곧바로 실제 악보 소프트웨어로 흘러갈 수 있게 했다. 또 합성+스캔 데이터셋, 베이스라인 모델, 표준 평가지표를 한 묶음으로 공개해 후속 연구의 비교 기준선을 마련했다. 이는 이후 Sheet Music Transformer, Sheet Music Benchmark 등 후속 OMR 연구의 토대가 되었다.

## 한계와 비판

1. **합성 데이터 의존**: 학습용 train 세트는 MuseScore로 렌더링한 깨끗한 합성 이미지뿐이고, 실제 스캔 데이터는 dev/test로만 제공된다. 따라서 모델이 합성 이미지의 깔끔한 글리프에 과적합되어, 잉크 번짐·기울어짐·낡은 인쇄 같은 실제 스캔의 노이즈에 취약할 수 있다(domain gap).

2. **시스템 단위 처리**: 모델은 악보 전체가 아니라 한 줄(시스템)씩 처리한다. 시스템을 잘라내는 전처리(세그멘테이션)가 별도로 필요하며, 페이지 전체를 통째로 끝-에서-끝으로 인식하는 진정한 의미의 end-to-end는 아니다.

3. **평가지표의 계산 비용**: TEDn은 음악적으로 타당한 지표지만 한 시스템당 최대 2분, 트리 크기에 제곱으로 증가해 대규모 평가가 매우 느리다. 점수 한 곡을 통째로 평가하는 것은 사실상 불가능하다고 저자들도 밝힌다.

4. **LMX의 의도적 정보 손실**: LMX는 시각적 표기에 집중하느라 다이내믹(셈여림), 헤어핀, 페달, 글리산도, 가사(lyrics), 일부 장식음·아티큘레이션을 의도적으로 버린다. 따라서 LMX→MusicXML 복원본은 원본 악보의 표현 정보 일부를 잃는다. "최소 손실"이라는 표현은 음표·박자·조표 등 핵심 골격에 한정된 말이다.

5. **장르 편향**: 데이터가 19세기 독일 가곡(Lieder) 피아노 반주에 치우쳐 있어, 현대 악보·재즈·복잡한 다성 푸가 등 다른 장르로의 일반화는 별도 검증이 필요하다.

이런 한계에도, "표준과 호환되는 선형 포맷 + 공개 데이터셋 + 표준 지표"라는 실용적 패키지를 제시한 점에서 이 논문은 피아노 OMR 분야의 중요한 기준점으로 평가된다.
